# TASK-260712: v0.1 release readiness

## asis

- MVP 受け入れ基準は全項目達成済みで、基本編集・VS Code keymap import・terminal capability fallback は動作する
- 一方で、日常利用を妨げる P1 の編集 gap と Linux 実機検証が残っている
- 配布はソース build のみで、release asset の生成・checksum・導入手順がない
- 製品名と配布方針が Open Question のまま README と ADR に残り、release gate が不明瞭だった

## tobe

- 製品名は `coda` とする。既存製品との同名リスクは許容し、利用者の混同が実害になった時点で再評価する
- v0.1 は macOS / Linux の GitHub Release asset として配布し、mise の GitHub backend から version 指定で導入できる
- release は `prepare` と人間専用の `publish` を分離する。Agent は build・検証まで行えるが、commit / tag / push / GitHub Release 作成は行わない
- v0.1 の必須項目と、release 後に回せる機能を明確に分離する

## v0.1 既知制限 (README / release notes 用メモ)

- wrap 中の `cursor.up/down` は logical line 移動 (VS Code の visual 移動と異なる。TASK-260711-18 notes 参照)
- read-only (10 MB 超) buffer は `file.saveAs` も含めて保存不可 (別名保存の escape hatch なし。要望が出たら解除を検討)
- read-only buffer では undo/redo も block される (`edit.*` 一括 block の一貫性優先)
- selection なしの `Tab` は literal tab 挿入 (VS Code の indent 動作とは異なる。selection ありなら indent)
- Linux binary は glibc 2.17+ 対象 (musl 静的 binary は未提供)
- 外部変更検知は save 時のみ (常時 watch なし)。conflict 時は 2 回目の save で強制上書き

## notes

### smoke test 結果 (260712, snapshot `release:prepare -- 0.1.0`)

- 全 4 asset の SHA-256 が個別 `.sha256` と一致。archive root に単体 binary `coda` のみ
- macOS x64: x86_64 mac host で展開 binary の `--version` = `coda 0.1.0` を確認
- Linux x64 (Docker):
    - Ubuntu 18.04 (glibc 2.27) — `--version` / `--help` / `keymap import vscode --print-report` が末端まで動作。import report 正常
    - CentOS 7 (glibc 2.17 ちょうど) — 起動 OK。glibc 下限が README 記載どおりであることを実証
    - 非 TTY 実行時は "Terminal capability: not detected; assuming modern" が出て editor 起動へ進む
    - PTY 経由の対話操作 (編集・保存・終了) と legacy capability fallback は container からの key 送り込みが安定しないため未実施 → 実機確認へ
- Linux arm64 / macOS arm64: build・checksum・archive 構造のみ確認 (実行環境なし)

### release 実装の学び

- goreleaser Rust builder は `cargo zigbuild` 前提だが、**apple-darwin target を zig linker に通すと SDK の system library (iconv 等) 解決が壊れる**。builds を darwin (`command: build` = system linker) / linux (zigbuild) に分割して解決
- zigbuild は target 無指定だと新しめの glibc symbol (2.28) を参照する。**`x86_64-unknown-linux-gnu.2.17` の suffix 指定で glibc 2.17 に pin** (最大要求 symbol が 2.17 になることを `strings | grep GLIBC` で確認)
- snapshot は tag が無いと version が `0.0.0-SNAPSHOT-*` になり asset 名検証ができないため、`snapshot.version_template` + `CODA_RELEASE_VERSION` env で prepare の version を注入
- zig の version 表示は `zig version` (`--version` 不可)。toolchain 検査の probe はツールごとに引数を変える

- `../gistan` の goreleaser + 二段階 release script を参照する。ただし Rust cross compile は linker / libc の差異があるため、Deno の target 指定を機械的にコピーしない
- Linux は glibc binary だけで「どこでも動く」とは言えない。v0.1 で musl を採用しないなら、README に最低 glibc version または検証 distro を記載する
- `coda` は検索性・商標・package namespace の面で強い名前ではない。「流行っていない」は安全性の根拠ではないため、利用拡大や配布先追加の前に再評価する
