# TASK-260712-19: v0.1 release readiness

## asis

- MVP 受け入れ基準は全項目達成済みで、基本編集・VS Code keymap import・terminal capability fallback は動作する
- 一方で、日常利用を妨げる P1 の編集 gap と Linux 実機検証が残っている
- 配布はソース build のみで、release asset の生成・checksum・導入手順がない
- 製品名と配布方針が Open Question のまま README と ADR に残り、release gate が不明瞭だった

## tobe

- 製品名は `coda` とする。既存製品との同名リスクは許容し、利用者の混同が実害になった時点で再評価する
- v0.1 は macOS / Linux の GitHub Release asset として配布し、mise の GitHub backend から version 指定で導入できる
- release は `prepare` と人間専用の `publish` を分離する。Agent は build・検証まで行えるが、commit / tag / push / GitHub Release 作成は行わない
- v0.1 の必須項目と、release 後に回せる機能を明確に分離する

## release scope（優先順）

### Gate 1: 日常利用を止める gap

- [ ] [TASK-260711-18](TASK-260711-18-wrap-toggle.md): wrap toggle と長行の視認性
- [ ] [TASK-260711-19](TASK-260711-19-small-ux-gaps.md): ファイルなし起動・設定ファイルを開く・indent / outdent
- [ ] Save As: 汎用 1 行 prompt と `file.saveAs`
- [ ] 外部変更検知: save 時の mtime 比較と上書き警告（常時 watch はしない）
- [ ] large file protection: 10 MB 超を read-only で開き、`isReadonly` context を実運用する

### Gate 2: 配布の再現性

- [ ] CI: GitHub Actions で `cargo fmt --check` / `cargo clippy -- -D warnings` / `cargo test` を macOS + Linux で実行する
- [ ] `.goreleaser.yaml`: macOS / Linux × x86_64 / aarch64 を build・archive 化し、asset ごとの SHA-256 checksum を生成する
- [ ] `scripts/release.ts`: `release:prepare` のみ Agent 実行可能にし、version bump・test・snapshot build をまとめる
- [ ] `scripts/release.ts`: `release:publish` は clean tree / tag / version を検証してから publish する人間専用操作とし、Agent が実行しない旨をコードと docs に明記する
- [ ] `mise.toml`: goreleaser と release task を定義する。Rust target / linker の不足は prepare 時に明示的に失敗させる
- [ ] README: source build、GitHub Release、mise による導入、手動 release 手順、対応 platform を記載する

### Gate 3: release candidate 検証

- [ ] Linux 実機で起動・編集・保存・終了・VS Code import・legacy capability fallback を確認する
- [ ] macOS Intel または cross-built x86_64 asset の起動可能性を確認する
- [ ] snapshot archive を展開し、binary が archive root にあり `coda --version` が tag version と一致することを確認する
- [ ] `mise use -g github:yano3nora/coda@<version>` で asset 自動判定と導入を確認する
- [ ] v0.1 release の既知制限を README / release notes 用メモにまとめる

## v0.1 から外す項目

以下は実用開始や配布を止めない。release 後の優先 backlog とする。

- `:which-key`、config.toml 残項目、import の `--cmd=keep|ctrl|both`
- split views、mouse support、keymap verify、他 editor importer
- Homebrew / crates.io / mise registry 登録、tag 起点の自動 publish、SSH bootstrap script

## testcases

- [ ] `mise run pre-commit` が成功する
- [ ] `mise run release:prepare -- 0.1.0` が push / publish せず全 target の archive と checksum を生成する
- [ ] version・tag・working tree が不整合な場合、publish helper が GitHub API 呼び出し前に失敗する
- [ ] README の install command と実際の asset 名が一致する
- [ ] macOS / Linux の smoke test 結果を本 TASK の notes に記録する

## notes

- `../gistan` の goreleaser + 二段階 release script を参照する。ただし Rust cross compile は linker / libc の差異があるため、Deno の target 指定を機械的にコピーしない
- Linux は glibc binary だけで「どこでも動く」とは言えない。v0.1 で musl を採用しないなら、README に最低 glibc version または検証 distro を記載する
- `coda` は検索性・商標・package namespace の面で強い名前ではない。「流行っていない」は安全性の根拠ではないため、利用拡大や配布先追加の前に再評価する
